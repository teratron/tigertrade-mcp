# Security

## Threat Model

`tigertrade-mcp` runs entirely on the user's local machine and brokers no network traffic to the outside world by itself. Its threat surface is therefore narrow but not zero. Concretely:

| Surface                          | What it touches                                                                                  | Risk if compromised                                                                  |
| -------------------------------- | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------ |
| MCP stdio transport              | A single child process spawned by the MCP client (Claude Code, Cursor, etc.) over `stdin/stdout` | Whatever the spawning client is allowed to do                                        |
| Tiger API WebSocket              | `ws://localhost:7819` exposed by Tiger Trade                                                     | Read chart state, switch the active symbol in a link-group (UI-only, no trading)     |
| Broker connector (Phase 2+)      | The broker's local terminal or API endpoint (e.g. Quik, Transaq, MT5, Rithmic)                   | Whatever permissions the broker credentials grant — including real orders            |
| Trading audit log                | `logs/trading-audit-*.jsonl` on the local filesystem                                             | Disclosure of order intent if the file is exfiltrated                                |
| Indicator SDK IPC (Phase 5)      | A local named pipe between the indicator DLL loaded by Tiger Trade and the server process       | Tampering with chart-data the server returns to the LLM                              |

The server **does not** open outbound network connections to anything outside `localhost` by itself. The MCP client may make external calls (e.g. Claude Code calling Anthropic's API), but that traffic is not generated by this server.

## Local-Only Network Model

All transports used by this server are local:

- MCP stdio — no socket at all.
- Tiger API — `ws://localhost:7819`, the loopback interface.
- Broker connectors — bind to the broker's local terminal (Quik, Transaq) or use the broker's API endpoint **only with credentials the user provides**.
- Indicator IPC — Windows named pipe (`\\.\pipe\tigertrade-mcp`), local-only.

The server **must not** open any listening TCP socket. If a contribution proposes a network listener (HTTP, gRPC, raw TCP, etc.), it will be rejected.

## Credential Handling

Broker credentials (API keys, passwords, certificate paths) are configuration, not code:

- **Never** commit credentials. The `.gitignore` blocks `.env`, `appsettings.*.Development.json`, `appsettings.Local.json`, and certificate file patterns.
- **Preferred storage**: environment variables with prefix `TTMCP_` (e.g. `TTMCP_Broker__Settings__ApiKey`).
- **Acceptable storage**: `appsettings.Development.json` next to the binary, gitignored.
- **Not acceptable**: hardcoded values, files in the repo, or values inlined in `.mcp.json`.

Logs scrub credentials. If you find a code path that logs a secret, treat it as a security bug and report it (see below).

## Trading Safety

The Phase 3 trading layer has three independent guards. **All three must be defeated** to fire a real order:

1. **Feature flag.** Without `--enable-trading` on the server command line, write-tools (`broker_place_order`, `broker_cancel_order`, `broker_modify_order`) are not registered with the MCP server at all. The LLM cannot call what does not exist.
2. **`dry_run: true` default.** Every write-tool parameter `dry_run` defaults to `true`. A dry-run logs the intent to the audit log but never reaches the broker.
3. **`confirm` string.** Every write-tool also requires an explicit `confirm` parameter with a literal acknowledgement text configured in `appsettings.json:Trading.RequireConfirmText`. The default is `"I-CONFIRM-LIVE-ORDER"`. The server rejects calls with any other value.

Plus an audit log: every attempt — including dry-runs and rejections — appends a structured line to `logs/trading-audit-*.jsonl`.

If you are modifying trading code, the [`AGENTS.md`](AGENTS.md) completion checklist requires you to verify all three guards remain intact. PRs that weaken these guards "for testing convenience" will be rejected; use `BrokerMock` instead.

## Tiger.com EULA Compliance

The Tiger.com EULA, Article 4.3.1, prohibits reverse engineering. This project enforces that prohibition as an architectural rule:

- No reflection against `TigerTrade.Api.dll` or any other proprietary Tiger Trade binary.
- No decompilation of Tiger Trade binaries.
- No use of undocumented APIs.
- Integration only through documented extension points (Tiger API ws@7819, Indicator SDK, external broker APIs).

See [`EULA-DISCLAIMER.md`](EULA-DISCLAIMER.md) for the full statement.

## Reporting a Vulnerability

If you discover a security issue, **please do not file a public GitHub issue**. Instead:

- Email the maintainer at `alexandrovoleg.ru@gmail.com` with the subject prefix `[tigertrade-mcp security]`.
- Include: affected version (commit SHA), reproduction steps, and what you believe the impact is.
- Allow up to 14 days for an initial response. Coordinated disclosure is appreciated.

In-scope:

- Credential leakage in logs, error messages, or transport.
- Trading-safety bypasses (any path that places a live order without the three guards).
- Local-network expansion (any listener bound to a non-loopback interface).
- Memory or input-validation issues in the MCP / WebSocket / broker connector code paths.

Out of scope:

- Issues in upstream dependencies (please report to the respective project; we will track and patch).
- Anything that requires the attacker to already have local code execution on the machine running the server (that attacker can do far worse than abuse this tool).
- Anything in Tiger Trade itself — report to Tiger.com directly.

## Dependency Hygiene

Direct NuGet dependencies are pinned in `*.csproj` files. CI (Phase 0 task T0.6) runs `dotnet list package --vulnerable` against the restored graph and fails on known vulnerable transitives. When a vulnerability is reported in a dependency, expect a patch release within one working week of upstream fix availability.
